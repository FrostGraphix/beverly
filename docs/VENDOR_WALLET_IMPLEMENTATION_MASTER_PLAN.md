# Production Wallet Platform Master Plan

Status:
- Production implementation plan.
- Replaces the earlier vendor-wallet planning draft.
- Root `ARCHITECTURE.md` remains canonical for repo-wide ownership.
- This document owns the wallet platform flow, logic, rollout, and Beverly CRM connection.

## 1. Product Boundary

Beverly CRM remains the parent system.

The CRM owns:
- staff authentication
- staff session ownership
- shared customer, meter, station, and tariff data
- token-generation access
- shared audit context
- the single sidebar entry named `Vending Wallet`

The wallet platform owns:
- vendor onboarding and verification
- wallet admin operations
- vendor self-service operations
- funding approvals
- wallet ledger and holds
- purchases, receipts, reversals, disputes, statements, reports, and reconciliation

The wallet must not become a second CRM. It is a connected financial subsystem with separate admin and vendor workspaces.

## 2. Workspace Flow

### Staff Flow

1. Staff logs into Beverly CRM.
2. Staff clicks `Vending Wallet`.
3. The app opens `#/wallet/admin/dashboard`.
4. No second staff login is shown.
5. Wallet admin routes trust the same backend session and role claims.
6. Backend authorization still checks the role before every wallet API action.

Wallet Admin pages:
- Dashboard
- Vendors
- Create Vendor
- Users & Roles
- Verification
- Wallet Balances
- Funding Approvals
- Manual Credits
- Vending Monitor
- Reversals
- Disputes
- Settlement
- Reports
- Audit Log

### Vendor Flow

1. Vendor opens the dedicated vendor auth page.
2. Vendor signs in with invite credentials or a temporary password.
3. If the password is temporary, the vendor must change it before portal access.
4. Vendor enters the vendor wallet workspace.
5. Vendor can view balance, fund wallet, buy token, send remotely, view transactions, retrieve receipts, view statement, and manage profile.
6. Vendor cannot enter CRM internal pages or wallet admin pages.

Vendor pages:
- Vendor Login
- Forced Password Change
- Dashboard
- Wallet Balance
- Top-up Request
- Buy Token
- Remote Send
- Transactions
- Receipts
- Statement
- Profile

## 3. Roles and Claims

Staff roles:
- `super-admin`
- `account`
- `finance-checker`

Vendor roles:
- `vendor_user`
- `vendor_manager`

Required claims:
- `role`
- `user_id`
- `vendor_organization_id` for vendors
- `wallet_status`
- `onboarding_status`
- `password_reset_required`

Rules:
- Staff roles can access Wallet Admin routes only.
- Vendor roles can access Wallet Vendor routes only.
- Vendor users are scoped to their own `vendor_organization_id`.
- Vendor payloads cannot choose another `organizationId` or `walletId`.
- Authenticated wallet writes derive actor identity from backend session, not request payload.
- Staff may enter vendor flows only through explicit impersonation/testing authority.

## 6. Funding Flow

1. Vendor creates a top-up request.
2. System generates funding reference and payment instructions.
3. Vendor uploads proof.
4. Proof is stored privately.
5. Finance reviews vendor identity, proof, duplicate hash, amount, channel, and prior funding history.
6. Approval posts immutable `funding_credit` ledger entry.
7. Rejection stores reason and keeps proof history.

Funding states:
- initiated
- proof_uploaded
- under_review
- approved
- rejected
- expired
- cancelled

## 7. Vending Flow

1. Vendor searches customer or meter using Beverly backend data.
2. Vendor previews tariff, amount, units, charges, delivery mode, and settlement path.
3. Backend creates purchase order and wallet hold atomically.
4. Vend engine dispatches token generation or remote-send.
5. Confirmed success captures the hold and creates a receipt.
6. Failure releases the hold.
7. Unknown delivery remains `delivery_pending_review`.
8. Receipt is always recoverable from history.

Purchase states:
- created
- hold_active
- dispatching
- delivered
- delivery_pending_review
- failed
- reversed

Delivery states:
- token_generated
- remote_send_success
- remote_send_pending
- delivery_failed
- delivery_unknown

## 8. Operations Flow

Wallet Admin monitors:
- funding approvals
- manual credits
- purchase failures
- pending remote-send
- reversals
- disputes
- frozen wallets
- settlement windows
- reconciliation mismatches
- role changes
- audit events

Manual credit rules:
- maker-checker required
- same actor cannot make and approve
- approval posts `manual_credit`
- rejection records reason
- no direct balance edit is allowed

Reversal rules:
- use compensating ledger entries
- preserve original purchase and receipt
- record support or finance reason

## 9. API Contract

Wallet APIs remain lowercase:
- `/api/vendor/*`
- `/api/wallet/summary`
- `/api/wallet/funding/*`
- `/api/wallet/purchase/*`
- `/api/wallet/manual-credit/*`
- `/api/wallet/reconciliation/*`

Rules:
- material writes require idempotency keys
- response statuses must be stable
- backend ignores client-supplied actor identity when authenticated
- vendor requests are scoped to session organization
- staff breadth is role-gated

### Phase 6: Vendor Wallet UX

### Phase 7: Backend Financial Hardening

- Make order and hold creation atomic.
- Add stable operation idempotency keys.
- Add reversal, dispute, hold expiry, proof hash, risk events, and audit services.
- Harden RLS and cross-tenant denial.

### Phase 8: Reporting, Reconciliation, and Monitoring

- Add settlement reports, statements, funding aging, failed purchase report, pending delivery report, frozen wallet report, and role audit report.
- Add stale hold sweeper, expired funding sweeper, anomaly detection, and reconciliation logs.

### Phase 9: Release Gates

- Require unit, API, RLS, browser, visual, build, and preview smoke tests.
- Verify CRM-to-wallet SSO, vendor auth separation, role denial, funding approval, token purchase, receipts, reversals, and audit trail.
- Defer customer-direct purchase until vendor wallet is stable.

## 12. Acceptance Criteria

The wallet platform is production-grade when:
- CRM remains the parent system
- staff enters Wallet Admin without re-auth
- vendors use a separate auth page
- temporary passwords force change before portal access
- vendor accounts require verification before wallet activation
- role matrix and role actions are audited
- all wallet tables are functional
- funding approval posts immutable ledger credit
- token purchase uses hold then capture
- failed purchase releases value
- receipts are always retrievable
- vendor RLS and backend guards block cross-tenant reads
- finance can reconcile daily operations without SQL shortcuts
