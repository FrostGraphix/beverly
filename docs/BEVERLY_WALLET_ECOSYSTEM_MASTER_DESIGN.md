## 1. Mission

Cut all direct public contact with the CRM.

Beverly CRM becomes back-office only.

Vendors get a retail bank-style portal.

Customers self-register and self-serve.

CRM staff approve vendors and run operations.

Customers cannot become vendors.

Vendors cannot self-onboard.

Every money move is ledger-derived.

Every token is generated by one engine.

Every receipt is immutable.

## 2. Three-Workspace Architecture

Three separate frontends. One backend platform. One database.

### 2.0 CRM ↔ Wallet Boundary (Hard Rule)

Wallet admin pages live in the **Wallet system only**.

Wallet admin routes live in the **Wallet system only**.

The CRM does NOT host wallet admin pages.

The CRM has exactly **ONE** sidebar entry pointing to the wallet:

- Label: `Wallet`
- Action: opens the wallet workspace (`vendor.beverly.com` admin surface or shared `wallet.beverly.com`).
- No nested wallet routes inside the CRM sidebar.
- No wallet sub-menu inside the CRM.
- No wallet tables, wallet forms, wallet dashboards inside the CRM.

Why:
- Keeps CRM focused on operations.
- Keeps wallet a true subsystem.
- Prevents drift between two surfaces.
- Single source of truth for wallet UX.

How staff transitions:
- Staff clicks `Wallet` in CRM sidebar.
- App opens wallet workspace in same window (or new tab).
- Session token is shared via SSO (Supabase Auth).
- Wallet workspace authorizes staff role on its own.
- No re-login.
- Return path: wallet header has `Back to CRM` link.

## 3. Identity Domains

Three identity stores. One auth provider (Supabase Auth).

| Domain | Audience | Self-signup | 2FA |
|--------|----------|-------------|-----|
| `admin.beverly.com` | CRM staff | NO | mandatory |
| `vendor.beverly.com` | Approved vendors | NO | mandatory |
| `app.beverly.com` | Public customers | YES | optional, encouraged |

Auth scopes are isolated.

A customer cannot log into vendor portal.

A vendor cannot log into admin portal.

Staff impersonation is logged and reversible.

JWT claims:
- `role`
- `actor_type`: `staff` | `vendor_user` | `customer`
- `actor_id`
- `vendor_organization_id` (vendor only)
- `customer_id` (customer only)
- `mfa_verified`
- `password_reset_required`
- `kyc_status` (customer / vendor)

Sessions:
- Staff: 30 min idle, 8 hour absolute.
- Vendor: 30 min idle, 12 hour absolute.
- Customer: 60 min idle, 30 day absolute (refresh token rotation).

## 4. Customer Self-Service Layer (NEW)

### 4.1 Goal

Every customer can:
- Register with phone or email.
- Verify identity (NIN or BVN).
- Onboard one or more meters they own.
- Buy electricity tokens for those meters.
- Buy a new meter (request + payment).
- View history, balances, receipts.
- Fund a personal wallet (optional).
- Get notifications for low credit, recharge success, debt.

### 4.2 Registration Flow

```
Step 1: phone/email + password
Step 2: OTP verification (SMS or email)
Step 3: KYC: NIN or BVN (Nigeria)
Step 4: Optional profile: name, address, station
Step 5: Land on Dashboard
```

OTP rules:
- 6-digit numeric.
- 5-minute expiry.
- Rate limit: 3/min, 10/hour per number.
- Lock after 5 failed verifications.

KYC rules:
- NIN via NIMC API.
- BVN via NIBSS or paystack identity API.
- Tier 1: phone-verified only → token purchase up to ₦50,000/day.
- Tier 2: NIN-verified → up to ₦500,000/day.
- Tier 3: BVN + address → unlimited.

### 4.3 Meter Onboarding Flow

```
Step 1: Customer enters meter number.
Step 2: System validates meter in CRM master data.
Step 3: System fetches customer-on-record from energy backend.
Step 4: Ownership verification (one of):
        a. OTP to registered phone (if energy backend has it)
        b. Last token / last recharge amount challenge
        c. Manual review (admin approves within 24 hours)
Step 5: Meter linked to customer account.
Step 6: Notifications enabled.
```

Multiple meters per customer allowed.

One meter can be linked to ONE customer at a time.

Transfer flow exists: original owner must release, new owner verifies.

### 4.4 Token Purchase Flow

Two purchase modes:

**Wallet mode:**
1. Customer selects meter.
2. Enters amount.
3. System validates tariff.
4. System computes units.
5. Customer confirms.
6. Wallet debited (atomic).
7. Token generated.
8. Token delivered: SMS + in-app + email.
9. Receipt available.

**Direct-pay mode:**
1. Customer selects meter.
2. Enters amount.
3. System computes units.
4. Customer initiates payment.
5. Payment gateway processes.
6. Webhook confirms.
7. Token generated.
8. Token delivered.
9. Receipt available.

Failure handling:
- Payment success + token failure → automatic wallet credit (refund) within 5 min.
- Stuck token → support queue.
- Customer never loses money on failure.

### 4.5 Meter Purchase Flow (NEW)

Customer can request a new meter.

```
Step 1: Select meter type (single-phase / three-phase / smart).
Step 2: Enter installation address.
Step 3: System assigns to nearest station.
Step 4: Payment via gateway.
Step 5: Order enters `meter_orders` table.
Step 6: CRM staff sees order in admin queue.
Step 7: Field technician dispatched.
Step 8: Meter installed and provisioned in energy backend.
Step 9: Meter auto-linked to customer.
Step 10: Customer notified.
```

Order states:
- `paid`
- `assigned`
- `dispatched`
- `installed`
- `activated`
- `cancelled`
- `refunded`

### 4.6 Customer Wallet (Optional)

Some customers prefer prefunding.

Wallet model:
- One wallet per customer.
- Balance is ledger-derived.
- Funding via gateway (card, transfer, USSD).
- Spend on token purchases or meter orders.
- Withdrawals not supported (closed-loop).
- Refunds return to wallet, not card (closed-loop).

Customer wallet states match vendor wallet states for code reuse.

## 5. Vendor Retail Layer (refined)

### 5.1 Strict Rule

Vendors cannot self-onboard.

Vendor self-signup endpoint does not exist.

A public "Vendor Interest" form exists. It writes to `vendor_applications` and notifies CRM staff. It does NOT create accounts.

### 5.2 Vendor Onboarding Flow

```
Step 1: Vendor submits public interest form (or staff invites directly).
Step 2: Staff reviews application in CRM.
Step 3: Staff requests documents (CAC, ID, address proof, bank).
Step 4: Vendor uploads via secure invite link (no account yet).
Step 5: Staff verifies documents.
Step 6: Staff approves OR rejects.
Step 7: On approval: system creates organization, user, wallet, temp password.
Step 8: Vendor receives credentials via approved channel.
Step 9: Vendor forced to change password and set up 2FA.
```

- Remote-send tokens to meters.
- View customers in their assigned region only.
- View their own transactions.
- View their own receipts.
- View settlement statement.
- Reprint receipts.
- Raise disputes.
- Manage sub-users (vendor_manager role only).

### 5.4 Vendor Cannot

- Modify customer master data.
- Modify meter master data.
- Modify tariff.
- Approve own funding.
- Approve own manual credit.
- See other vendors' data.
- See staff or admin pages.

## 7. Payment Gateway Integration

### 7.1 Providers

Primary: **Paystack** (dominant Nigerian gateway, NGN-native).

Fallback: **Flutterwave** (multi-currency, redundancy).

Direct bank transfer with proof upload remains for vendors.

USSD payment via Paystack USSD code for customer wallet top-up.

### 7.2 Payment Flow Pattern

```
1. Frontend calls /api/payment/initiate
   → backend creates payment_transaction (status=initiated)
   → backend generates idempotency_key
   → backend calls gateway init API
   → returns redirect_url or inline reference

2. Customer pays via gateway.

3. Gateway calls /api/payment/webhook
   → backend verifies HMAC signature
   → backend checks idempotency_key
   → backend updates payment_transaction (status=succeeded|failed)
   → backend posts wallet ledger credit (if funding)
   → backend triggers token generation (if direct purchase)
   → backend writes audit event

4. Frontend polls /api/payment/status
   → returns final state
   → token revealed only after success

5. Backup: cron sweeps stuck payments every 5 min.
```

### 7.3 Critical Rules

- Webhook signature ALWAYS verified.
- Replay attacks blocked by `gateway_reference` uniqueness.
- Frontend cannot mark payment successful.
- Backend trusts only webhook + verify-API.
- Reconciliation job nightly: gateway records vs DB.

### 7.4 Card Storage

Never store PAN/CVV.

Gateway tokenization only.

Save token in `payment_methods` table:
- `id`
- `customer_id`
- `gateway` (paystack|flutterwave)
- `token` (gateway-issued)
- `last4`
- `brand`
- `expiry`

PCI-DSS handled by gateway (SAQ-A scope).

### 7.5 Refunds

Closed-loop:
- Customer refund → customer wallet credit.
- Vendor refund → vendor wallet credit.
- Gateway refund triggered ONLY for unsuccessful service delivery.
- Refund must be approved by finance role (maker-checker).

## 9. Wallet Ledger Architecture

### 9.1 Universal Ledger

One ledger table serves vendors AND customers.

```sql
wallet_ledger_entries (
  id uuid primary key,
  wallet_id uuid not null,
  wallet_owner_type text not null,  -- 'vendor' | 'customer'
  wallet_owner_id uuid not null,
  direction text not null,           -- 'credit' | 'debit'
  amount_minor bigint not null,      -- kobo
  balance_after_minor bigint not null,
  entry_type text not null,          -- enum below
  reference_type text,               -- 'funding_request' | 'purchase_order' | 'meter_order' | 'reversal'
  reference_id uuid,
  idempotency_key text unique,
  memo text,
  created_by uuid not null,
  created_at timestamptz not null default now()
)
```

### 9.2 Entry Types

| Entry Type | Direction | Trigger |
|-----------|-----------|---------|
| `funding_credit` | credit | Approved funding |
| `payment_credit` | credit | Gateway payment success |
| `purchase_debit` | debit | Token purchase |
| `meter_order_debit` | debit | Meter purchase |
| `manual_credit` | credit | Admin credit (maker-checker) |
| `manual_debit` | debit | Admin debit (rare, audit-heavy) |
| `reversal_credit` | credit | Compensating entry |
| `reversal_debit` | debit | Compensating entry |
| `fee_debit` | debit | Service fee |
| `promo_credit` | credit | Promotional credit |

### 9.3 Invariants

- Entries are **immutable**. Never UPDATE, never DELETE.
- Corrections via compensating entry.
- `balance_after_minor` computed in DB function with row-level lock.
- Atomic via Postgres transaction.
- Idempotency key prevents double-post.

### 9.4 Balance Derivation

```sql
SELECT balance_after_minor
FROM wallet_ledger_entries
WHERE wallet_id = $1
ORDER BY created_at DESC, id DESC
LIMIT 1;
```

Materialized view `wallet_balances` refreshed on every insert via trigger.

Reconciliation job nightly: sum entries vs cached balance.

### 9.5 Holds

```sql
wallet_holds (
  id uuid primary key,
  wallet_id uuid not null,
  amount_minor bigint not null,
  reference_type text,
  reference_id uuid,
  status text not null,    -- 'active' | 'captured' | 'released' | 'expired'
  expires_at timestamptz,
  created_at timestamptz
)
```

Available balance = ledger balance - sum of active holds.

Hold → capture writes debit entry.

Hold → release deletes hold (no ledger entry, no movement happened).

Hold expiry: 15 minutes. Sweeper cron releases.

## 10. Token Generation Pipeline (Unified)

### 10.1 Single Engine

CRM token engine remains canonical.

All three paths route through `token-flow.mjs`:
- CRM staff manual generate.
- Vendor purchase.
- Customer self-purchase.

This guarantees one source of truth for tariff, units, tax.

### 10.2 Pipeline

```
Request → validate customer + meter + tariff
       → compute units (kWh) from amount and effective price
       → call energy backend POST /api/token/creditTokenRecord/create
       → receive token + metadata
       → store in purchase_orders
       → store receipt payload
       → emit notification (SMS, email, in-app)
       → return token to caller
```

### 10.3 Token Delivery

For meter-resident tokens:
- Display in UI.
- SMS to registered phone.
- Email to registered email.
- In-app notification.
- Persist in receipt.

For remote-send tokens:
- Energy backend remote-send API.
- Poll until delivered or failed.
- Customer never enters token manually.
- Notification on delivery.

### 10.4 Failure Modes

| Failure | Recovery |
|---------|----------|
| Tariff lookup failure | Block purchase, do not charge |
| Backend timeout | Mark `pending`, retry 3x with backoff |
| Backend error | Mark `failed`, refund wallet |
| Token issued but display lost | Recoverable from `purchase_orders.token` field |
| Remote-send fail | Mark `delivery_failed`, refund or retry |

## 13. Security Model

### 13.1 Authentication

- Supabase Auth as single provider.
- Email/phone + password.
- Magic-link option for customers.
- Mandatory 2FA: staff, vendors.
- Optional 2FA: customers (TOTP via authenticator app, SMS fallback).
- Forced password rotation: 90 days for staff/vendors.
- Password rules: 12+ chars, uppercase, lowercase, number, symbol, no breach list (HIBP).

### 13.2 Authorization

- Row Level Security (RLS) enforced on every table.
- Customer policies: see own rows only.
- Vendor policies: see organization rows only.
- Staff policies: role-gated.
- RLS bypassed only by trusted server-side functions with explicit grants.

### 13.3 RLS Examples

```sql
-- customer wallet RLS
create policy customer_own_wallet
on wallets for select
to authenticated
using (
  wallet_owner_type = 'customer'
  and wallet_owner_id = (auth.jwt() ->> 'customer_id')::uuid
);

-- vendor purchase RLS
create policy vendor_own_purchases
on purchase_orders for select
to authenticated
using (
  vendor_organization_id = (auth.jwt() ->> 'vendor_organization_id')::uuid
);
```

### 13.4 Rate Limiting

| Endpoint | Limit |
|----------|-------|
| Login | 5/min/IP |
| Signup | 3/hour/IP |
| OTP request | 3/min, 10/hour/phone |
| Token purchase | 10/min/customer |
| Wallet fund | 5/min/customer |
| Public vendor application | 1/day/IP |
| Webhook | 100/min/source |

Implementation: Redis + sliding window (Upstash).

### 13.5 Input Validation

- Zod schemas on every endpoint.
- SQL injection: parameterized queries, no raw SQL with user input.
- XSS: HTML escape in render, CSP headers.
- CSRF: SameSite=Lax cookies, double-submit token for state-changing requests.
- File upload: size limit (10MB), MIME validation, virus scan via ClamAV.

### 13.6 Secrets

- Never in code.
- Never in client bundles.
- Vercel environment variables (production).
- `.env.local` (development).
- Rotation: quarterly. Documented in `TOKEN_ROTATION_RUNBOOK.md`.
- Vault: Doppler or 1Password Connect (optional upgrade).

### 13.7 Transport

- HTTPS only.
- HSTS: 2 years, includeSubDomains, preload.
- TLS 1.3 minimum.
- HTTP→HTTPS redirect at edge.

### 13.8 Encryption

- At rest: Supabase managed (AES-256).
- Sensitive PII fields (NIN, BVN, last4): app-level AES-256-GCM, key in Vercel env.
- Receipts: signed URLs, no public bucket.

### 13.9 Fraud Signals

For customer payments:
- Velocity check: >5 purchases/hour flag.
- Geo anomaly: payment from new country.
- Device anomaly: new device + new IP.
- Amount anomaly: >3σ above customer baseline.
- BIN check: prepaid card on high-amount.
- Chargeback list match.

Action on high signal:
- Step-up auth (re-verify OTP).
- Manual review queue.
- Auto-block if score >90.

### 13.10 Vendor Anti-Fraud

- Daily purchase cap.
- Suspicious meter pattern: same meter >X times/day.
- Geo IP vs registered region.
- Sub-user anomaly detection.

Mobile-first.

PWA-capable.

Offline-tolerant for receipt history.

### 15.3 Delivery Rules

- Retry 3x with exponential backoff.
- Dead-letter queue after final failure.
- User can opt out per channel per event (except mandatory: OTP, security).
- Quiet hours: 22:00–06:00 (non-urgent only).
- Template-driven, stored in `notification_templates`.

### 15.4 Compliance

- SMS opt-in tracking.
- Unsubscribe footer on email.
- DPR compliance for Nigerian users.
- Audit log of all sent messages.

## 16. Fraud & Risk Engine (Beyond EIH)

EIH fraud engine remains for meter-side fraud (tamper, divergence, etc).

New: **Transaction fraud engine** for payment-side.

### 16.1 Real-Time Scoring

On every purchase, compute risk score:

| Signal | Weight |
|--------|--------|
| New device | 10 |
| New IP | 5 |
| New geo | 15 |
| Velocity (>5/hr) | 20 |
| Amount > 3σ baseline | 15 |
| Card BIN risk | 10 |
| Meter mismatch with customer | 25 |

Score 0–100. >70 triggers step-up auth. >90 blocks.

### 16.2 Outputs

Writes to `fraud_signals`.

Admin sees aggregate in `/admin/fraud`.

False positives reviewed weekly.

### 17.1 Audit Coverage

- Export all data (GDPR-style).
- Request deletion (with regulatory hold exemption).

| Fraud baseline recompute | 05:00 daily | Update customer baselines |
| Stuck purchase scan | every 10 min | Flag pending >30min |

- Auto-deploy preview on PR.
- Auto-deploy production on merge.

### 19.4 Database Migrations

- Supabase CLI migrations.
- Forward-only.
- Reviewed via PR.
- Run in staging first.

Flagsmith or PostHog feature flags.

## 23. Testing Strategy

### 23.1 Pyramid

- Unit (Vitest): pure functions, services, ledger logic.
- Integration (Vitest + Supabase test DB): API endpoints, RLS policies.
- E2E (Playwright): critical flows in three browsers, mobile viewports.
- Visual regression (Percy or Chromatic): UI surface stability.
- Load (k6): payment + token flow at expected peak.

### 23.2 Critical Test Scenarios

Must pass before production:
- Customer signup → KYC → meter link → token purchase.
- Vendor onboarding → funding → vending → receipt.
- Concurrent purchases on same wallet (no negative balance).
- Webhook idempotency (replay safe).
- RLS denial on cross-tenant access.
- Token engine failure → automatic refund.
- Gateway failure → fallback to secondary.
- Hold expiry frees balance.
- Reconciliation matches.
- 2FA enforcement.

### 23.3 Test Data

- Faker for synthetic data.
- Anonymized prod snapshots in staging.
- No PII in tests.
- Seeded test customers, vendors, meters.

## 24. Acceptance Criteria

The ecosystem is production-ready when:

- CRM has no public surface.
- CRM sidebar has exactly one `Wallet` link and no nested wallet routes.
- All wallet admin pages render only in the wallet workspace.
- All three domains have separate auth.
- Customer can sign up and onboard a meter without staff help.
- Customer can buy tokens via gateway, get token in <60 sec.
- Vendor cannot self-onboard.
- Vendor must be approved by staff.
- Vendor can fund and vend.
- All money moves through immutable ledger.
- All receipts are retrievable.
- All payments reconcile.
- All audit events are written.
- RLS prevents all cross-tenant access.
- 2FA enforced on staff and vendor accounts.
- Notifications deliver >99% within 30 sec.
- Reconciliation passes daily.
- Fraud engine flags suspicious purchases.
- Backup and restore drilled monthly.
- Penetration test passed.
- NDPR and CBN compliance signed off.
- Observability dashboards green.
- All critical alerts have runbooks.
